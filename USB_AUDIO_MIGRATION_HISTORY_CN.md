# CH32H417 USB 音频专用例程改造记录

## 1. 文档目的

本文档用于完整记录本工程从“键盘鼠标 Host 示例”整理为“USB 音频专用例程”的全过程，包括：

- 目标是什么
- 遇到了哪些问题
- 每个问题的现象、根因和解决办法
- 关键代码改动落在哪些文件
- 改造过程中踩过哪些坑
- 当前工程最终收敛成了什么结构

本文档重点不是做功能说明，而是保留整个工程演进过程，便于后续：

- 回顾设计决策
- 理解为什么代码会长成现在这样
- 后续继续扩展 USB Audio Host 功能时避免重复踩坑

## 2. 改造目标

初始目标并不是做一个通用 USB Host，而是要让 CH32H417 官方 `Host_KM` 例程：

1. 识别 USB 有线耳机
2. 支持使用开发板的 Type-C 接口接入耳机
3. 在串口输出中明确看到耳机枚举过程
4. 支持耳机线控按键触发测试
5. 支持录音和回放
6. 最终从“键盘鼠标例程”收敛成“USB 音频专用例程”
7. 代码结构清晰，适合 USB 音频初学者阅读

## 3. 初始工程状态

改造前的工程基础是官方 `Host_KM` 例程，特点如下：

- 工程主线是键盘/鼠标/HUB 的 Host 示例
- 应用层文件名是 `app_km.c` / `app_km.h`
- 根设备和 HUB 下级设备的主服务路径默认围绕键盘、鼠标和通用 HID 展开
- 对 USB Audio 设备没有完整识别和运行时支持
- Type-C 口虽然在硬件上连接到 USBFS/USBPD，但示例代码没有先建立 Type-C Source 角色

也就是说，最开始这个工程并不具备“插上 Type-C 数字耳机就能工作”的基础条件。

## 4. 第一阶段：让工程能识别 USB 音频耳机

### 4.1 问题：音频耳机被识别成 Unknown

#### 现象

耳机插入后，没有像键盘那样出现清晰的类型输出，或者只能看到枚举前半段，无法确认它是否被归类为音频设备。

#### 根因

原始示例的类型分析逻辑更偏向简单设备，默认假设：

- 配置描述符后面很快就是标准接口描述符
- 设备类型判断不需要完整遍历整份配置描述符

但很多 USB 耳机是复合设备，常见结构是：

- Audio Control 接口
- 播放 Audio Streaming 接口
- 录音 Audio Streaming 接口
- 线控 HID 接口
- 可能还会带类专用描述符和 IAD 结构

如果只看局部描述符，很容易把设备判成未知类型。

#### 解决办法

重写设备类型分析逻辑，完整遍历配置描述符中的所有接口描述符，按接口类进行归类：

- 发现 Audio 接口则归为 `USB_DEV_CLASS_AUDIO`
- 发现 HUB 接口则归为 `USB_DEV_CLASS_HUB`
- 其余保持 Unknown

#### 涉及文件

- `Common/USB_Host/app_km.c`，后续重命名为 `Common/USB_Host/app_audio.c`

#### 结果

能够在枚举阶段正确识别 USB Audio 设备，并输出：

- 设备 VID/PID
- 每个接口的 Class/SubClass/Protocol
- 最终的设备类型判断结果

---

### 4.2 问题：耳机插入后没有类似键盘那样的串口输出

#### 现象

USB 键盘插入时串口会持续看到数据，而耳机即使被插入，也没有明显输出，用户容易误以为“没有识别”。

#### 根因

这是两个不同层次的问题：

- 键盘示例有持续的 HID 轮询和数据输出
- 音频设备在最初阶段只做了枚举识别，没有持续打印音频设备摘要

所以它不是“没识别”，而是“日志不够明显”。

#### 解决办法

补充音频设备摘要输出：

- 枚举成功后打印设备 VID/PID
- 打印所有音频接口信息
- 打印播放流、录音流、HID 控制接口

#### 结果

只要设备成功枚举，串口就能明确看到：

- `Root Device Is Audio(Headset)`
- `Audio IF:...`
- `Play IF:...`
- `Mic IF:...`
- `Ctrl IF:...`

## 5. 第二阶段：确认 Type-C 耳机为什么完全没有输出

### 5.1 问题：Type-C 耳机插入 Type-C 口时，串口完全没有输出

#### 现象

耳机插在板子的 USBFS Type-C 口时：

- 串口没有 `USB Port Dev In.`
- 没有设备枚举日志
- 代码侧看起来像“根本没检测到设备”

但耳机插到电脑上能识别为 `AB13x USB Audio`。

#### 根因

这一步不是协议问题，而是 Type-C 口角色建立问题。

官方硬件设计里，USBFS Type-C 口除了 `D+` / `D-` 外，还有：

- `CC1` / `CC2`
- `VBUS_CC`

数字 Type-C 耳机要真正上线，主机侧必须满足两个条件：

1. 在 `CC1/CC2` 上提供 Source 角色的 `Rp`
2. 在 `VBUS` 上提供稳定 5V

原始 `Host_KM` 例程只初始化了 USBFS Host，没有初始化 USBPD/Type-C Source 角色，因此耳机根本不会开始 USB 枚举。

#### 解决办法

在硬件初始化里增加 Type-C Source 初始化：

- 使能 USBPD 相关时钟
- 配置 `PB3/PB4` 为 `USBPD_CC1/CC2`
- 在 `CC1/CC2` 上拉 `Rp`
- 确保后续 USBFS Host 初始化前，Type-C 口先具备 Source 角色

另外，实验中还确认了一点：

- 如果 `VBUS_CC` 没有被接到 5V，耳机仍然不会上线

#### 涉及文件

- `Common/hardware.c`

#### 结果

当 `VBUS_CC` 被正确接到 5V 后，串口开始出现：

- `USB Port Dev In.`
- `Enum:`
- `Get DevDesc:`
- `Get CfgDesc:`

这说明 Type-C 口的上电和角色建立问题已经打通。

## 6. 第三阶段：确认耳机的真实结构

### 6.1 问题：需要确认这只耳机究竟是什么设备

#### 现象

用户提供的枚举日志显示这只耳机的配置描述符较长，包含多个接口，必须先读懂设备拓扑，后面才能继续做录音回放。

#### 枚举结果分析

从实际描述符可以确认这只设备是一个复合音频耳机，包含：

1. `IF0`：Audio Control
2. `IF1`：Audio Streaming（耳机播放）
3. `IF2`：Audio Streaming（麦克风录音）
4. `IF3`：HID（线控按键）

进一步可以确定：

- 播放端点：`EP 0x03 OUT`
- 录音端点：`EP 0x83 IN`
- 线控端点：`EP 0x82 IN`
- 采样率：48kHz
- 播放格式：双声道、16bit
- 录音格式：单声道、16bit

#### 结论

这不是“只有输出没有输入”的耳机，而是完整的：

- 扬声器
- 麦克风
- 线控 HID

所以从协议角度，工程具备继续做录音回放测试的条件。

## 7. 第四阶段：从“能识别”推进到“能打开音频流”

### 7.1 问题：只枚举成功还不够，音频流没有真正启动

#### 现象

虽然设备被识别为音频耳机，但：

- 没有实际录音数据
- 没有实际播放声音

#### 根因

对 USB Audio 设备来说，枚举结束只代表“设备存在”。

要真正让播放/录音流工作，还必须：

1. 通过 `SET_INTERFACE` 切到带音频端点的 `AltSetting`
2. 对采样率可控的端点发送 `SET_CUR Sampling Frequency`
3. 必要时通过 `Feature Unit` 取消静音、设置音量

原始示例根本没有这些步骤。

#### 解决办法

新增音频流控制路径：

- 为播放流和录音流实现统一的 `USBH_AudioSetStreamState()`
- 在启流时完成：
  - `SET_INTERFACE`
  - `SET_CUR Sampling Frequency`
  - `Feature Unit` 的静音/音量初始化
- 在停流时切回 `Alt0`

#### 涉及文件

- `Common/USB_Host/app_audio.c`
- `Common/USB_Host/ch32h417_usbfs_host.c`
- `Common/USB_Host/ch32h417_usbfs_host.h`
- `Common/USB_Host/usb_host_config.h`

#### 结果

串口可以看到：

- `Set IF1 Alt1: OK`
- `Set IF2 Alt1: OK`
- `Set Play Fs:48000 OK`
- `Set Mic Fs:48000 OK`

说明设备已经接受了播放流和录音流的启动命令。

## 8. 第五阶段：打通线控按键

### 8.1 问题：线控按键按了没反应

#### 现象

耳机的线控按键按下时，没有触发预期的“开始录音 / 停止录音 / 播放录音”。

#### 根因

一开始工程只知道设备带一个 HID 接口，但并不知道：

- HID 报告是什么格式
- 哪个字节代表按下
- 初始报告是否会误触发状态机

用户后续提供的实际日志显示：

- 按下：`Headset HID: 10`
- 释放：`Headset HID: 00`

#### 解决办法

线控路径最终做了这些处理：

1. 读取 HID Report Descriptor，用于确认接口存在
2. 轮询 HID 中断输入端点
3. 对收到的报告做“变化检测”
4. 把第一包作为初始基线，不触发动作
5. 只在 `00 -> 非零` 的“按下沿”触发状态机
6. 增加约 300ms 去抖

#### 结果

线控按键最终可以稳定驱动状态机：

- 开始录音
- 停止录音
- 回放录音

## 9. 第六阶段：录音流为什么一开始总是 0 包

### 9.1 问题：开始录音后，停止时总是 `0 bytes, 0 packets`

#### 现象

状态机已经工作正常，但每次停止录音时：

- `record done, 0 bytes, 0 packets`

#### 根因

问题不在线控按键，而在底层等时传输状态判断。

音频录音端点是等时 `IN` 端点，不走普通控制/Bulk/中断那种 ACK/NAK 语义。原始主机栈更多是为普通端点写的，所以：

- 等时端点完成后没有被正确视为成功
- 主机一直没有把麦克风数据包计入成功包数

#### 解决办法

修正 USBFS Host 底层等时传输处理：

- 调整 `USBFSH_Transact()` 对等时端点返回状态的成功判定
- 让等时 IN/OUT 端点使用正确的事务模式
- 避免把等时端点误按普通握手端点处理

#### 涉及文件

- `Common/USB_Host/ch32h417_usbfs_host.c`

#### 结果

录音包开始真正累积，录音结束时可以看到非零 `bytes` 和 `packets`。

## 10. 第七阶段：播放流为什么没有声音、只有蜂鸣或时长异常

### 10.1 问题：开始能发声，但不是录音，是蜂鸣，且播放时长不对

#### 现象

随着播放链路逐步打通，出现过几种异常：

- 没声音
- 一长段蜂鸣
- 播放时长和录音时长不一致
- 播放卡在错误日志里

#### 根因一：播放 OUT 端点的事务模式和结果判定不对

在不同阶段，播放端点曾出现：

- `ERR_USB_TRANSFER`
- `NAK`
- 首包后一直失败

这说明等时 `OUT` 的状态处理和数据 PID 翻转逻辑一开始并不匹配这只耳机。

#### 根因二：USBFS 包缓冲大小不够

耳机实际包尺寸是：

- 播放包约 192 字节
- 录音包约 96 字节

但底层曾经还保留过更小的缓冲配置，导致：

- 数据截断
- 异常蜂鸣
- 时长错乱

#### 根因三：录音缓存长度统计被截断

曾经录音缓存空间计算里存在 16 位截断，导致：

- 录音长度经常固定卡在某个错误值
- 回放时长明显不对

#### 解决办法

最终围绕播放链路做了几轮修正：

1. 修正等时 `OUT` 端点的事务结果判定
2. 修正数据 PID / toggle 处理策略
3. 把 USBFS 包缓冲大小增大到能覆盖音频包尺寸
4. 修正录音长度统计变量类型
5. 增加播放错误日志，定位 `Spk Tx Err`
6. 在播放路径里只在发送成功后推进播放位置

#### 涉及文件

- `Common/USB_Host/ch32h417_usbfs_host.h`
- `Common/USB_Host/ch32h417_usbfs_host.c`
- `Common/USB_Host/app_audio.c`

#### 结果

播放链路最终稳定到：

- 开机旋律可播放
- 录音回放可播放
- 播放时长与录音长度对应关系正常

## 11. 第八阶段：为什么需要开机先播放 5 秒旋律

### 11.1 问题：必须先区分“播放链路问题”还是“录音内容问题”

#### 现象

在录音回放阶段，如果听不到自己的声音，无法立刻判断问题出在：

- 播放链路没通
- 录音没录到
- 录音太小
- 音量/静音没解除

#### 解决办法

增加开机自动播放 5 秒旋律功能：

- 不依赖录音缓存
- 不依赖麦克风输入
- 只单独验证扬声器播放链路

#### 意义

这一步非常关键，它把问题拆成两半：

- 如果开机旋律都没声音，优先查播放链路
- 如果开机旋律有声音，但录音回放没声音，优先查录音内容和回放增益

#### 结果

开机旋律最终可播放，证明：

- Type-C 上电正常
- 播放流启用正常
- 喇叭 OUT 端点发包正常
- Feature Unit 基本可用

## 12. 第九阶段：录音格式与存储策略调整

### 12.1 问题：需要更长录音时间，但板上 RAM 有限

#### 现象

最初直接保存 PCM 时，录音时长受 RAM 限制较大，不适合做“任意时长录音，最大 10 秒”的测试。

#### 解决办法

改成：

- 录音阶段把麦克风 PCM 转成 `IMA ADPCM 4-bit`
- 回放阶段再实时解码为 PCM

#### 优点

- 同样 RAM 下可显著延长录音时长
- 适合当前“测试型音频主机例程”
- 对学习 USB 音频数据流也有帮助

#### 结果

工程最终支持：

- 任意时长录音
- 最大 10 秒
- 线控按键开始/停止录音
- 线控按键回放

## 14. 第十一阶段：从“通用示例”裁剪成“音频专用例程”

### 14.1 问题：虽然已经能工作，但代码里仍保留很多键盘鼠标示例遗留语义

#### 现象

如果不继续整理，会有这些问题：

- 读代码时容易把精力浪费在无关路径上
- `HOST_CTL` 中保留过多通用字段，不利于理解音频主线
- 文件名和模块职责不够聚焦

#### 解决办法

收敛工程结构：

1. 根口和 HUB 下级设备运行态只保留音频设备服务
2. 类型分析重点收敛为 `Audio / HUB / Unknown`
3. 删除或停用无关的键盘鼠标业务路径
4. 保留 HUB 和底层 USB Host 通用能力，因为音频耳机仍可能挂在 HUB 下

#### 结果

工程逻辑已经不是“键盘鼠标示例 + 音频补丁”，而是“音频专用例程 + 必要的 USB/HUB 基础设施”。

## 15. 第十二阶段：注释与编码问题

### 15.1 问题：中文注释出现乱码

#### 现象

在整理中文注释时，曾经出现：

- 中文显示成乱码
- 编辑器中看到一长串错误字符
- 一部分注释甚至影响到源码阅读

#### 根因

根因不是单纯“文件没保存成 UTF-8”，而是更早一次通过终端修改文件时发生了错误转码：

- 原始注释内容被损坏
- 后续即使再切编码，也无法自动恢复原文

#### 解决办法

最终采用了两步法：

1. 不再做自动转码
2. 把历史乱码注释整体删除，再按中文手工重写

并且明确要求：

- 注释全部写中文
- 编码使用 UTF-8
- 关键主线函数优先加详细说明

#### 结果

当前 `app_audio.c` 已完成一轮主线注释重写：

- 文件头说明
- 全局数据说明
- 定时器时基说明
- 音频枚举说明
- 运行态服务说明
- 主状态机说明

### 15.2 问题：恢复源码时使用 `.lst` 文件带来新的截断错误

#### 现象

在一次修复过程中，`app_audio.c` 曾被从 `.lst` 文件恢复，导致：

- 长行被截断
- 条件表达式只剩半截
- 函数声明断裂
- 字符串字面量不完整

进而引发多次编译错误，例如：

- 未知类型名
- 缺少分号
- `invalid storage class`
- 字符串未终止
- 成员名被截断

#### 根因

`.lst` 文件是汇编/列表输出，不适合当作稳定源码恢复基线。长行在列表文件里可能被截断或换行，恢复后并不能保证源码完整。

#### 解决办法

后续排查改为：

1. 先按编译器报错逐点修复断裂语句
2. 用结构扫描确认括号/花括号平衡
3. 修完结构后再回到注释整理
4. 不再把 `.lst` 当作长期可靠源码来源

#### 结果

`app_audio.c` 最终回到了可继续编译和维护的状态。

## 16. 第十三阶段：构建与静态检查问题

### 16.1 遇到的问题类型

在代码功能基本完成后，还清理过几类工程问题：

1. `printf` 格式化参数类型不匹配
   - `%lu` 对应参数是 `uint32_t`
2. `if` / `else` 缩进不一致
3. 局部长行恢复时缺少分号
4. 结构体成员名被截断
5. 端点描述符字段访问方式与实际头文件不匹配

### 16.2 解决办法

- 对 `uint32_t` 打印参数统一显式转成 `(unsigned long)`
- 重排 `if/else` 缩进，不改业务逻辑
- 对端点包长优先用原始字节解析，避免依赖不一致的结构体成员命名

## 19. 改造过程中最重要的经验

### 19.1 先分层定位，不要一开始就盯协议细节

本次排障最有效的思路是分层判断：

1. 有没有 `USB Port Dev In.`
2. 有没有 `Get DevDesc` / `Get CfgDesc`
3. 有没有 `Audio IF` / `Play IF` / `Mic IF`
4. 录音端有没有数据包
5. 开机旋律有没有声音
6. 线控 HID 有没有变化

这种分层方法能快速判断问题在：

- 硬件供电
- Type-C 角色
- USB 枚举
- 音频流启停
- 等时传输
- 音量/静音
- 录音数据内容

### 19.2 Type-C 数字耳机首先是 Type-C 问题，其次才是 USB Audio 问题

如果没有 `CC Rp` 和 `VBUS 5V`，耳机甚至不会开始 USB 枚举。这个阶段改再多音频代码都没有意义。

### 19.3 开机旋律是排障分层的关键测试手段

只要开机旋律有声音，就说明：

- 播放链路已经成立
- 录音回放问题可以只聚焦录音内容和回放增益

### 19.4 `.lst` 不适合当源码恢复基线

以后如果再出现源码损坏，优先：

- 从真实源文件备份恢复
- 从版本管理恢复
- 或按编译报错逐步修补

不要直接把 `.lst` 反向当作源码长期使用。
